Docs-Driven Development, practiced

Software whose source is documentation. Humans write docs at the abstraction they choose; AI agents make reality match them. This page renders ddd-practice — one specific way to adopt and operate the idea — for fast human reading.

The idea in one picture

A programming language forces you to decide everything, including the thousands of things you have no opinion about. Docs as source invert that: write precisely where behavior matters, stay abstract where it doesn't. Natural language becomes a programming language with an adjustable determinism dial:

Pinned"An agent at zero balance never thinks." One reading. Behavior fixed.
Constrained"Respond within a minute under normal load." A family of correct implementations.
Open"The fleet should feel alive." Direction. Implementers decide the rest.
The rules (from the idea)
  • Minimal docs. Only what was explicitly specified. Everything unwritten is delegated, not forgotten. Reference the original instead of duplicating; only implementation may duplicate.
  • ./docs holds the docs, clearly separated from everything derived. The README is itself derived: a faithful summary of what the repo is and contains, linking into ./docs — it specifies nothing new and misdescribes nothing.
  • Docs are the source of truth. Code exists to satisfy them; where they disagree, the code is wrong.
  • To change the software, change the docs. Code is then made to conform.
  • Ambiguity below the written precision is freedom, not error.
  • Divergence that matters means the docs were too loose exactly there — the fix is precision exactly there.

Adopting an existing project

An existing project is an implementation whose docs don't exist yet. Three moves; an afternoon of work; the third starts a loop that never ends.

Declare. State in the governing docs (they live in ./docs; create the root document if nothing qualifies) that docs govern: docs are truth, code is derived, behavior changes start as doc changes. Link the idea so the words have a definition. The README becomes a derived summary. The declaration changes what every later divergence means — before it, a disagreeing doc is stale; after it, the code is buggy.
Bootstrap. Build the hierarchy from three sources: existing docs, the code as it behaves, and the owner's intent — the one input only the owner can give. Promote existing fragments instead of duplicating them. A pre-existing ./docs is raw material, not a fait accompli: rework what genuinely governs into the hierarchy; move everything descriptive (research, notes, generated output) elsewhere in the repo, described abstractly in the docs if it matters. Owners write intent; agents draft everything else — an owner's pass turns description into specification.
Reconcile. Run the first conformance audit immediately, not eventually. It will find things: undocumented behavior others depend on, documented behavior never built, invariants quietly dropped. Every finding makes the docs truer or the code correct — both are adoption working.

Operating: every change starts in the docs

Find the governing sentence that owns the behavior you want to change.
Change the doc first. Silent docs + internal change = your freedom, just build. Silent docs + a boundary others depend on = write the missing sentence first.
Conform the code — same commit as the doc change at the latest, never after. Evolve or rewrite is purely a cost question.

Code-only behavior changes are violations, even good ones. A fix worth keeping is worth a doc sentence.

The conformance loop

Periodically, a party other than the code's author reads the docs fully, lists every checkable claim, and audits the implementation. Self-graded conformance is how "docs-driven" degrades into "docs-adjacent". Each divergence is classified — and the class decides which side gets fixed:

Code bug
The doc is right; the code fails it.
→ fix the code
Doc bug
The code is sensible; the doc is stale or wrong.
→ fix the doc
Ambiguity
Two plausible readings; the code picked one.
→ tighten the one sentence — nothing more
Unspecified
Significant behavior the docs never mention.
→ others depend on it: write it in (or the owner removes it) · internal: leave it, that's freedom

The auditor also reports what verified as conforming, so trust in the docs is earned, not assumed.

Layers: precision lives at boundaries

Intentwhy this exists — short, stable, mostly open
Contractsinterfaces, protocols, invariants others depend on — the precision lives here
Component detailonly where detail matters; many components need nothing beyond their contracts

conflicts resolve upward: the higher layer wins, the lower doc gets fixed

Ambiguity inside one component is free — nobody else depends on those choices. Ambiguity at an interface is where two correct implementations collide. When integration breaks, the failure names the sentence that was too loose; tighten that sentence and nothing else.

Promotion: how the docs learn

When a bug fix or incident encodes a real behavioral rule, promote the rule into the docs at its layer. Code remembers nothing across a rewrite; docs remember everything. A rule living only in code is one regeneration away from being lost. Each divergence event ends with a corrected artifact or a tightened sentence — and the tightened sentence prevents the class, not just the instance.

Talking to the human

The human reads docs, not code. Report in docs and observable behavior: which claims held, which were violated and how it showed, what changed in which document. Decisions that change intent arrive as proposed doc edits — here is the sentence, here is what it would become, here is why. The human's answer is an edit, and the edit is the decision. Everything below the abstraction the human chose is not their problem; that is the point.
